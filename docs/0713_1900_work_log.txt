1. 프로젝트 관리 화면 분석 버튼 흐름 수정

- 문제:
  - 마이페이지에서 프로젝트 관리 화면으로 바로 들어가면 채용공고 등록 정보(jobPostingId)가 없었다.

4. 채용공고 이미지 업로드 입력 추가

7. URL 크롤링 정보 부족 감지 추가

- 문제:
  - 잡코리아 같은 URL은 실제 공고 본문을 충분히 긁지 못할 수 있는데, 기존 로직은 80자 이상 텍스트만 있으면 성공으로 처리했다.
  - 결과적으로 네비게이션/푸터/부족한 본문만 긁힌 경우에도 분석이 진행될 수 있었다.

- 처리:
  - URL로 긁은 텍스트를 구조화한 뒤 requiredSkills와 preferredSkills가 모두 비어 있으면 "핵심 자격요건을 충분히 읽지 못함"으로 판단하게 했다.
  - 새 에러 코드 JOB_POSTING_URL_INSUFFICIENT를 추가했다.
  - 프론트에서는 이 에러를 받으면 이미지 업로드 탭으로 전환하고, 공고 화면 캡처 이미지 또는 직접 입력을 안내한다.
  - FastAPI의 에러 응답 구조(detail.error.code)도 프론트 ApiError가 읽을 수 있게 client.ts를 수정했다.

- 관련 파일:

8. "근거 보기" 데이터 의미 수정

- 문제:
  - 기존 "근거 보기"는 /evidences/{id} API로 README 원문 덩어리를 그대로 가져와 보여줬다.
  - 하지만 사용자가 기대하는 근거는 "왜 이 프로젝트가 이 공고와 맞는지"인데, README 원문은 보기도 불편하고 매칭 근거로서 논리적으로 약했다.

- 처리:
  - 백엔드 analysis 응답의 recommendedProjects에 matchEvidence 필드를 추가했다.
  - matchEvidence에는 다음 정보를 담는다.
    - requirement: 공고 요구사항 또는 매칭 대상
    - matchType: skill | semantic | missing
    - source: 프로젝트 기술 스택, 프로젝트 요약 등
    - projectEvidence: 프로젝트에서 확인된 근거 문장
    - explanation: 왜 이 항목이 매칭 또는 보완 필요인지 설명
  - 프론트 EvidenceAccordion은 더 이상 /evidences/{id}로 README 원문을 불러오지 않는다.
  - 이력서 초안의 프로젝트 섹션은 해당 프로젝트의 matchEvidence를 보여준다.
  - 요약/스킬 섹션은 추천 프로젝트들의 주요 matchEvidence를 모아서 보여준다.

- 관련 파일:
  - backend/app/routers/analysis.py
  - frontend/src/types/analysis.ts
  - frontend/src/components/EvidenceAccordion.tsx
  - frontend/src/components/EvidenceAccordion.css
  - frontend/src/pages/RecommendationResultPage.tsx

- 이미지 OCR/vision 분석은 OPENROUTER_API_KEY와 이미지 입력을 지원하는 OPENROUTER_MODEL이 필요하다.
